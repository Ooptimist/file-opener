# 拖拽功能问题复盘与经验总结

本文档记录本项目在“拖拽文件到当前选择文件框”功能上的问题、根因、解决方案和后续建议，供下次开发参考。

## 1. 现象与问题

- 现象 1：拖拽后文件没有出现在“当前选择的文件”列表。
- 现象 2：拖拽后程序偶发闪退。
- 现象 3：某些修复方案在脚本自测通过，但真实 Explorer 拖拽场景不稳定。

## 2. 根因分析

### 2.1 旧方案（WndProc 子类化）风险高

- 使用 Win32 `WM_DROPFILES` + `SetWindowLongPtrW` 子类化窗口过程时，容易出现：
  - 64 位参数签名不一致导致异常。
  - 目标控件命中判断不稳定（控件尺寸/坐标时机问题）。
  - 消息重复处理或 `DragFinish` 生命周期处理不当，导致崩溃风险。

### 2.2 “只在文件框生效”带来的判定复杂度

- 仅允许目标区域生效时，需要准确命中“目标控件或其子控件”。
- 如果目标选得不对（例如选到高度异常的内部控件），会误判为“框外”。

### 2.3 自测方式与真实场景差异

- 直接模拟消息（`SendMessage`）只能覆盖部分链路。
- 真实 Explorer 拖拽涉及 shell 侧行为、权限(UIPI)、tk 事件循环时序等，可能与模拟环境不同。

## 3. 最终稳定方案

## 3.1 技术路线切换

- 不再继续使用 WndProc 子类化方案作为主路径。
- 改为 `tkinterdnd2` 的 Tk 级拖拽事件方案（`<<Drop>>`），新增独立模块：
  - `src/handlers/tkdnd_drop_zone.py`

## 3.2 集成方式

- 主应用类继承：`TkinterDnD.DnDWrapper + ctk.CTk`
- 初始化时调用：`TkinterDnD._require(self)`
- 仅对目标区域控件注册拖拽：
  - `target_widget.drop_target_register(DND_FILES)`
  - `target_widget.dnd_bind("<<Drop>>", handler)`

## 3.3 行为一致性

- 拖拽与“选择文件”按钮统一走同一添加逻辑（去重、过滤、刷新），确保最终效果一致。

## 4. 本次关键经验

- **经验 1：优先选择稳定的框架级能力**
  - 在 GUI 场景中，优先用 `tkdnd` 这类成熟事件机制，少做底层 WndProc 黑魔法。

- **经验 2：拖拽目标要选“稳定容器”而不是可能高度异常的内部控件**
  - 目标控件建议使用外层容器（例如 `file_list_container`）。

- **经验 3：统一业务入口，避免双逻辑漂移**
  - 按钮添加与拖拽添加必须走同一方法，减少分叉 bug。

- **经验 4：先 build 再验收**
  - 用户验证的是 `dist/FileOpener.exe`，每次修复后应先执行 `build.bat` 再测试。

- **经验 5：模拟测试不能替代真实拖拽测试**
  - 模拟消息用于回归，但最终必须做真实 Explorer 拖拽验收。
